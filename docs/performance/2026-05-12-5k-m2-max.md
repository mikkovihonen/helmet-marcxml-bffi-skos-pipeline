# 5 000-record run, M2 Max 64 GB, 2026-05-12

First end-to-end production-style run after P-02 closed. The
sample size (5 000) was chosen as the smallest slice that
exercises every stage of the pipeline (M1-M10) with real
escalate-band pairs while still finishing in one working
session.

### Where the time goes

```
M9 reconcile  5722 s ████████████████████████████████████  82.9 %
M3 bf-to-bffi  473 s ███                                    6.9 %
M6 judge       258 s ██                                     3.7 %
M2 marc-to-bf  206 s █                                      3.0 %
M5 embed       155 s █                                      2.2 %
M8 merge        44 s                                        0.6 %
load            22 s                                        0.3 %
skosify         16 s                                        0.2 %
```

**M9 reconcile is the new bottleneck after P-02.** M6 took 258 s;
M9 took 22× as long. The reason is fan-out: M6 ran 49 LLM calls
(one per escalate-band pair), but M9 runs **one LLM picker call
per ambiguous authority-control field on every canonical Work**.
For 3 157 canonical Works with multiple creators / subjects each,
the call count explodes well beyond M6's 49.

## M6 throughput vs the P-02 § A6 prediction

| Metric | A6 prediction (steady-state on M2 Max c=4) | Actual on this run |
|---|---|---|
| Pairs/min | 31.2 | ~14.9 (64 / 4.3 min) |
| Median latency / call | 1.94 s (warm) / 7.73 s (c=4 batched) | 3.2 s |

Two reasons the run came in below the steady-state ceiling:

- **Cold prefix cache** at run start. After server restart the
  first ~10 calls pay the full prefill; the A6 prediction is for
  warmed-up state.
- **Sample size is small.** 64 calls is too few to amortise the
  per-call overhead the bench script bypasses (langchain wrap,
  Pydantic post-validation, SQLite cache lookup).

On a longer-running corpus (50 k+ pairs) the M6 wall-time should
converge to the A6 ceiling.

## Extrapolation to the full 800 k corpus

Scaling each stage linearly from the 5 000-record results:

| Stage | 5k | 800k (linear extrap.) | Notes |
|---|---|---|---|
| M2 | 206 s | ~9.2 h | XSLT + boundary-1/2 gates; CPU-bound, parallelisable |
| M3 | 473 s | ~21 h | Linear in record count |
| M5 | 155 s | ~7 h | Linear in FAISS index size; memory-bound at scale (3.3 GB index for 800k × 1024-dim) |
| M6 | 258 s | ~11 h | Escalate-band sub-linear because most pairs reject early; if the 0.98 % escalate rate holds, ~7 800 escalate pairs at the A6 31/min ceiling → 4-5 h |
| M8 | 44 s | ~2 h | Merge is fast |
| **M9** | **95 min** | **~10 days** | **Doesn't fit overnight.** Number of LLM calls scales with canonical-Work × authority-fields-per-Work, both of which grow with corpus size. |
| skosify | 16 s | ~45 min | Fast |
| load | 22 s | ~1 h | Network-bound |

Headline: **a full-corpus run on this stack is M9-bound, not
M6-bound.** Production extrapolation needs an M9 strategy
(prompt-prefix caching like P-02 Phase B did for M6? batched
picker? cache-on-authority-key?) before the 800k batch is
practical.

## Bonus findings — six fixes shipped during this run

The 5k run surfaced six pipeline-robustness issues, each fixed
in its own commit. A re-run after these fixes would recover
~22 of the 666 dropped records (and proportionally ~3 500 on
the full corpus):

| Finding | Commit | Recovered |
|---|---|---|
| `JudgeCache` SQLite cross-thread `InterfaceError` at `M6_CONCURRENCY > 1` | [`1452a4f`](https://github.com/mikkovihonen/helmet-marcxml-bffi-skos-pipeline/commit/1452a4f) | unblocks M6 batched mode |
| marc2bibframe2 emits `xml:lang="ru-"` (trailing hyphen) on `008 / 041` Russian / Ukrainian records | [`2c832f6`](https://github.com/mikkovihonen/helmet-marcxml-bffi-skos-pipeline/commit/2c832f6) | 7 |
| Sierra exports subfields with `code=" "` (whitespace) | [`2033469`](https://github.com/mikkovihonen/helmet-marcxml-bffi-skos-pipeline/commit/2033469) | 1 (and ~half of the empty-datafield class below) |
| Sierra exports empty datafields (`<datafield ... />` with no subfields) | [`c235bf4`](https://github.com/mikkovihonen/helmet-marcxml-bffi-skos-pipeline/commit/c235bf4) | 2 |
| M2 SHACL shape over-reaches into marc2bibframe2's 856-derived secondary `bf:Instance`; `_find_root_resources` non-deterministically stamps the wrong Instance | [`549baa0`](https://github.com/mikkovihonen/helmet-marcxml-bffi-skos-pipeline/commit/549baa0) | 10 |
| BIBFRAME 856 modelling as `bf:Instance` vs `bf:Item` (deeper semantic question) | proposal [`prop-07`](../plans/proposed/prop-07-bibframe-856-as-item.md); workaround in `549baa0` | tracked for future |

The remaining failure categories from the run:

- **646 `marcxml-content-minimum` failures** (missing `1XX/7XX` creator). These are a *deliberate* spec gate; the
  pipeline can't anchor a Work to a missing creator. Whether
  to relax that gate (mint anonymous Works) is the conversation
  driven by [`prop-05`](../plans/proposed/prop-05-anonymous-work-canonicalisation.md).
- **3 `marcxml-xsd-validation` failures** beyond what the
  Sierra-export fix recovers — they're cataloguer-side data-
  quality issues that need source correction.
