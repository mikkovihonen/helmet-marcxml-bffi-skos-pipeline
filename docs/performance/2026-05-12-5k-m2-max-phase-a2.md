## Phase 1 sub-linear scaling — why `phase1=8` didn't give 8× on Phase 1

Decomposing the wall against Phase A's breakdown (Phase 1 serial ~3 800 s, Phase 2 c=4 ~1 660 s):

| | Phase 1 wall | Phase 2 wall | Total |
|---|---|---|---|
| Phase A (phase1=1, c=4) | ~3 800 s | ~1 660 s | 5 460 s |
| **Phase A2 (phase1=8, c=4)** | **~1 980 s** | ~1 660 s | 3 639 s |
| Speedup on Phase 1 | **~1.92×** | — | — |

`phase1=8` delivered ~1.9× speedup on Phase 1 — significantly less than the linear 8× a perfect scaling model would predict. The likely culprits, in approximate order of impact:

1. **Per-request server-side latency dominates throughput.** Finto's REST API + VIAF both respond in ~200–500 ms per query mostly on server-side compute / index lookup; concurrent requests pay server-side queue time rather than running in parallel on the server.
2. **`@lru_cache` hit rate is corpus-shape-dependent and serialises some queries.** `_finto_search_query`'s LRU caches repeated literals; the serial Phase A path benefited from those hits naturally, the concurrent Phase A2 path can still benefit but the lru get/put dispatches sequence under the GIL.
3. **Default `httpx.Client` connection pool** (`max_connections=100, max_keepalive_connections=20`) is more than enough for `phase1=8` — *not* a constraint.
4. **Fuseki SPARQL queries are local and fast** (~50 ms); their concurrent execution is bound by Fuseki's worker pool, but at `phase1=8` we're well below typical limits.

The sub-linearity is genuine but the absolute gain (~31 min wall saved) is the meaningful result. Pushing `phase1=16` would likely yield diminishing returns; the next lever is to *reduce* Phase 1 work, not parallelise it harder — i.e. Phase C (tier-0 normalisation expansion).

## Implications for Phase B, Phase C, and beyond

- **Phase B (persistent picker cache)** — projected to save up to the ~1 660 s of Phase 2 picker wall on warm reruns. Even at 100 % hit rate that brings the total to ~2 000 s (~33 min); enough to hit the ≥ 3× cumulative target if nothing else changes. **Worth shipping.**
- **Phase C (tier-0 normalisation + altLabel inclusion)** — pushes entities out of Phase 1's Finto/VIAF candidate-query branch into tier-0 local SPARQL. Both branches are now parallelised (Phase A2), but tier-0 is ~5× faster per query than Finto HTTP, AND a tier-0 hit avoids the picker call too. **Highest leverage per LOC of any remaining phase.**
- **Phase 1 / Phase 2 overlap** — currently the orchestrator runs Phase 1 to completion, *then* Phase 2. Streaming Phase-2-eligible entries into the picker pool as soon as they're known (without waiting for the slowest Phase 1 worker) would let Phase 1 and Phase 2 walls overlap. Saving potential: `min(Phase1, Phase2)` ≈ 1 660 s. This isn't in the current P-10 scope; flag for a follow-up plan if Phase B + C still don't hit the overnight target.
